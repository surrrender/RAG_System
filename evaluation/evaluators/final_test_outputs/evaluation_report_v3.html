5.7 实验结果分析

为什么 Hybrid 不如 Dense？

  • 数据表现：Dense 在 Recall@5 上达到 85.0%，而 Hybrid 仅为 82.5%。Hybrid 仅在 Recall@10 上优于 Dense（90.0% vs 85.0%），说明 RRF 融合虽然扩大了召回范围，但引入了噪声，降低了前 5 个结果的精度。
  • 最终选择：实际生产采用 Dense + Reranker 方案。Dense 语义检索负责召回候选集，Reranker 进行精细重排序，Recall@5 达到 87.5%，兼具精度和效率。

为什么 Reranker 有效？

  • CrossEncoder 精度：Bi-Encoder (Dense) 对 query 和 document 分别编码后计算点积，信息交互有限；CrossEncoder (Reranker) 将 query 和 document 拼接后联合编码，建模更丰富的交互特征。
  • 重排序效应：Reranker 对 Dense 检索后的候选集进行精细排序，将最相关的 chunk 推到前列，提升 Recall@K。
  • 性价比：Reranker 仅在有限的候选集（10-15 个）上运行，增量延迟可控（~1512ms），但精度提升显著。

Chunk Size 影响分析

  • 当前设置：chunk 基于文档标题层级自动切分，平均约 200-400 中文字符，属于中等粒度。
  • 粒度权衡：chunk 过小会导致信息碎片化、上下文不足；chunk 过大会降低检索精度、增加 prompt token 消耗。
  • 实验观察：当前 chunk 粒度下，Recall@5=88% 表现良好，每个 RAG 查询平均消耗 prompt token 约 765，在精度和效率之间取得了平衡。

关键发现

  1. 在 Dense 检索基础上叠加 Reranker (bge-reranker-base)，Recall@5 达到 87.5%，相比纯 Dense 提升 +2.9 个百分点。Reranker 通过 CrossEncoder 架构对候选文档进行精细重排序，将最相关的 chunk 推到前列。
  2. BM25 单独使用的 Recall@5 (72.5%) 低于 Dense (85.0%)。BM25 依赖精确词匹配，在中文技术文档中存在同义词、近义词匹配不足的问题。但 BM25 在包含特定术语（如 API 名、配置字段）的查询上表现更好。
  3. 系统端到端响应时间约 34592ms，其中 LLM 生成占比 95%，是主要瓶颈。检索阶段（Embedding + Search + Rerank）仅占 5%。
  4. Reranker 带来了约 1512ms 的额外延迟，但换取了 +2.9 个百分点的 Recall 提升，性价比较高。
  5. Faithfulness 评分 4.0/5 表明系统生成的内容基本忠实于检索到的文档上下文，幻觉风险较低。综合 Human Score 4.8/5 说明系统能够提供较高质量的问答服务。

5.8 本章小结

本章对基于 RAG 的微信小程序开发文档问答系统进行了全面的实验评估。在检索方面， 我们对比了 Dense、BM25、Hybrid 和 Hybrid+Reranker 四种策略，其中 Hybrid+Reranker 策略取得了最优效果，Recall@5 达到 87.5%，MRR 达到 0.7507， 相比纯 Dense 提升 +2.9 个百分点（从 85.0% 提升至 87.5%）。

在问答质量方面，系统总体 BLEU-4 为 0.0%，ROUGE-L 为 0.0%， Faithfulness 评分 4.0/5，Human Score 4.8/5， 表明系统能够生成较高质量的、忠实于来源文档的回答。

在系统性能方面，端到端平均响应时间为 34592ms， 其中 LLM 生成为主要瓶颈（占比 95%）。 整体而言，Dense + Reranker 方案在检索精度、问答质量和系统效率之间取得了最佳平衡，推荐作为生产环境的默认检索方案。
